# 0-3岁成长发育 H5 PRD v1.1

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 产品名称 | 0-3岁成长发育 H5 |
| 文档版本 | v1.1 |
| 文档状态 | 产品需求修订稿 |
| 目标端 | 移动端 H5，优先适配微信内访问 |
| 首版定位 | 面向 0-3 岁婴幼儿照护者，提供“当前月龄自动匹配阶段”和“跨阶段多维数据对比”的成长发育参考工具 |
| 核心原则 | 自动匹配、横向对比、来源可信、医学边界清晰、首版范围可落地 |

## 2. 背景与问题

0-3 岁婴幼儿成长发育信息通常分散在指南、PDF、表格、疫苗计划、体检建议和日常照护资料中。普通家长在实际照护时，最常见的问题不是“缺少资料”，而是很难快速判断当前宝宝处于哪个成长阶段，以及不同阶段之间的体重、身高、头围、睡眠、喂养、发育等数据如何变化。

本产品要优先解决两个问题：

- 自动匹配：用户输入宝宝出生日期后，系统自动计算当前月龄，并匹配到对应成长阶段。
- 多维对比：用户可以按维度对比不同阶段的宝宝参考数据，理解不同月龄阶段的变化趋势和重点差异。

产品只提供育儿参考和信息导航，不做医疗诊断，也不替代医生、儿保机构或接种单位的判断。

## 3. 产品目标

### 3.1 用户目标

- 用户可以通过宝宝出生日期自动匹配当前月龄和对应阶段。
- 用户可以快速查看当前阶段的核心生长指标和照护重点。
- 用户可以按体重、身高、头围、睡眠、喂养、发育、医疗保健等维度，对比不同阶段的数据和建议。
- 用户可以在“当前阶段详情”和“跨阶段对比”之间顺畅切换。
- 用户可以看到数据来源、版本、更新时间和免责声明。
- 用户发现数据或文字问题时，后续版本可以提交带上下文的反馈。

### 3.2 业务目标

- 将静态育儿资料转成结构化、可按阶段和维度双向组织的 H5 查询产品。
- 建立“宝宝当前阶段自动匹配”和“跨阶段维度对比”两条核心使用路径。
- 用数据来源和版本机制提升可信度。
- 后续用反馈闭环发现并修正数据问题。
- 在核心查询体验稳定后，再扩展分享、收藏和个性化传播能力。

### 3.3 成功指标

| 指标 | 口径 | 目标建议 |
| --- | --- | --- |
| 阶段匹配完成率 | 进入出生日期输入流程后成功匹配阶段的比例 | v1.0 达到 80% 以上 |
| 当前阶段信息触达率 | 首页访问后看到当前阶段核心指标的比例 | v1.0 达到 90% 以上 |
| 多维对比使用率 | 用户进入维度对比页或切换对比维度的比例 | 用于判断核心对比能力价值 |
| 阶段切换率 | 用户在对比页切换不同月龄阶段的比例 | 用于判断跨阶段浏览需求 |
| 详情页点击率 | 首页或对比页进入阶段详情、维度详情的比例 | 用于判断内容深度需求 |
| 搜索成功率 | 搜索后点击任一结果的比例 | 搜索上线后持续观察 |
| 反馈有效率 | 提交反馈中可复核、可处理的比例 | 用于评估反馈表单设计 |
| 数据修复周期 | 有效反馈从确认到发布修复的时间 | 建议控制在 7 天内 |

## 4. 范围规划

### 4.1 MVP 范围

MVP 只解决两个核心问题：

- 用户能否根据宝宝当前月龄自动匹配到正确阶段。
- 用户能否按多个维度对比不同阶段的宝宝参考数据。

| 模块 | 是否进入 MVP | 说明 |
| --- | --- | --- |
| 宝宝出生日期输入与阶段匹配 | 是 | 核心能力，必须 P0 |
| 首页当前阶段摘要 | 是 | 展示当前阶段、体重、身高、头围和重点提醒 |
| 多维度阶段对比 | 是 | 按维度对比不同阶段的参考数据 |
| 阶段详情 | 是 | 展示当前阶段完整内容，作为自动匹配后的落地页 |
| 维度详情 | 是 | 至少支持生长指标、睡眠、喂养、发育、医疗保健 |
| 数据说明与免责声明 | 是 | 医疗安全和可信度必需 |
| 基础来源标记 | 是 | 展示来源名称、页码或数据表 |
| 搜索 | 否 | 放 P1，不影响两条核心路径 |
| 反馈提交 | 可选 | 若已有处理机制则进入 MVP，否则放 P1 |
| 分享海报 | 否 | 放到传播版本 |
| 收藏、最近查看、反馈记录 | 否 | 放到增强版本 |

### 4.2 版本节奏

| 版本 | 目标 | 主要能力 |
| --- | --- | --- |
| v1.0 MVP | 跑通阶段匹配和多维对比 | 阶段匹配、首页、阶段详情、维度对比、维度详情、数据说明 |
| v1.1 增强版 | 提升查找和纠错效率 | 搜索、原文依据、反馈提交、筛选、数据版本检查 |
| v1.2 传播版 | 提升复访和传播 | 成长卡、分享海报、收藏、最近查看、反馈记录 |
| v1.3 运营版 | 支持内容持续维护 | 后台处理反馈、数据发布、修复通知、灰度更新 |

### 4.3 非目标

- 不判断宝宝是否“正常”或“异常”。
- 不提供诊断、治疗、用药、接种决策。
- 不替代儿保医生、接种单位或当地政策。
- 首版不建设完整账号体系。
- 首版不追求复杂社区、问答、课程或电商能力。

## 5. 用户与场景

| 用户类型 | 场景 | 产品策略 |
| --- | --- | --- |
| 新手家长 | 不确定当前月龄该看什么 | 首页直接给当前阶段摘要和重点提醒 |

| 传播型用户 | 想把宝宝当前阶段信息分享给家人 | 在后续版本提供成长卡和分享海报 |

## 7. 核心流程

### 7.1 阶段匹配流程

1. 用户进入首页。
2. 若本地没有宝宝出生日期，系统展示轻量设置入口，不阻塞基础浏览。
3. 用户输入出生日期。
4. 系统按当前日期计算宝宝年龄，输出月龄和天数。
5. 系统匹配到对应阶段。
6. 首页切换为个性化阶段摘要。

验收标准：

- 出生日期为空时，用户仍可手动选择阶段。
- 出生日期不能晚于当前日期。
- 超过 0-3 岁范围时，提示当前产品适用范围。
- 阶段匹配结果必须可手动调整。

### 7.2 多维度对比流程

1. 用户进入对比页。
2. 系统默认展示当前宝宝所处阶段，并同时展示相邻阶段或全部阶段的参考数据。
3. 用户选择对比维度，例如体重、身高、头围、睡眠、喂养、动作发育、语言发育、疫苗、体检。
4. 系统按阶段横向展示该维度的数据、建议或状态。
5. 用户可切换性别、阶段范围或数据类型。
6. 用户点击某个阶段或字段后，可进入阶段详情或维度详情。

验收标准：

- 对比页必须支持至少 3 个生长指标维度：体重、身高、头围。
- 对比页必须支持至少 3 个文本/建议类维度：睡眠、喂养、发育。
- 当前宝宝所在阶段必须高亮。
- 不同阶段的数据必须能在同一页面横向比较。
- 数据缺失时必须展示缺失说明，不能用空白代替。
- 生长指标必须支持男孩、女孩或全部筛选。

### 7.3 当前阶段查询流程

1. 用户进入首页。
2. 首页展示当前阶段名称、年龄范围和核心摘要。
3. 用户查看体重、身高、头围参考范围。
4. 用户进入阶段详情或维度详情查看完整说明。
5. 用户可从当前阶段进入跨阶段对比，理解该阶段在完整成长周期中的位置。
6. 用户可查看数据来源和免责声明。

验收标准：

- 首页首屏必须出现当前阶段和至少 3 个核心指标。
- 所有指标必须标注单位。
- 缺失数据不能空白展示，必须说明缺失原因。

### 7.4 维度浏览流程

1. 用户进入维度中心。
2. 用户选择生长指标、睡眠、喂养、发育、医疗保健等维度。
3. 系统展示该维度在不同阶段下的内容。
4. 用户可切换阶段或查看来源。

验收标准：

- 维度内容必须能回到阶段上下文。
- 生长指标类支持性别维度。
- 文本建议类必须展示适用阶段。
- 维度详情必须提供进入跨阶段对比的入口。

### 7.5 搜索流程

搜索建议放在 v1.1。若 MVP 数据量较小，可先使用分类入口替代搜索。

1. 用户输入关键词。
2. 系统匹配标题、分类、阶段、内容摘要和来源字段。
3. 搜索结果按相关性和阶段匹配度排序。
4. 无结果时展示推荐关键词和分类入口。

验收标准：

- 首版搜索可采用结构化字段匹配，不强制全文检索。
- 搜索结果必须能跳转到具体阶段或维度。
- 无结果不能只展示空白，需要给出替代入口。

### 7.6 反馈流程

反馈建议放在 v1.1，前提是存在后续处理机制。

1. 用户从指标、原文依据或数据说明页进入反馈。
2. 系统自动带入阶段、字段、当前值、来源页等上下文。
3. 用户选择问题类型并填写描述。
4. 系统提交反馈并返回反馈编号。
5. 运营或内容维护人员处理反馈，确认后进入数据修复流程。

验收标准：

- 反馈必须带上下文，不能只提交自由文本。
- 问题描述必填。
- 若没有反馈处理后台，首版只保留“反馈入口占位”或外部联系方式。

## 9. 数据治理

### 9.1 数据生命周期

| 阶段 | 要求 |
| --- | --- |
| 来源收集 | 记录来源文件、版本、发布日期、适用范围 |
| 结构化录入 | 拆分为阶段、维度、指标、来源字段 |
| 数据校验 | 检查单位、年龄范围、性别范围、来源页、空值说明 |
| 发布 | 生成数据版本号和更新时间 |
| 客户端加载 | 支持本地缓存和版本检查 |
| 反馈修复 | 有效反馈进入修正队列 |
| 回滚 | 发现严重错误时可回退到上一稳定版本 |

### 9.2 数据模型建议

| 实体 | 核心字段 |
| --- | --- |
| `stage` | id、name、start_day、end_day、summary |
| `growth_metric` | stage_id、metric_key、gender、min_value、max_value、unit、source_id、note |
| `text_content` | stage_id、category、title、content、source_id、risk_level |
| `comparison_dimension` | key、name、category、value_type、unit、supports_gender、sort_order |
| `comparison_item` | dimension_key、stage_id、gender、display_value、min_value、max_value、summary、source_id |
| `medical_item` | type、name、stage_id、age_label、content、source_id、region_note |
| `source` | id、source_name、page_no、section、excerpt、version、updated_at |
| `feedback` | id、context_type、context_id、issue_type、description、contact、status、created_at |
| `data_version` | version、changes、published_at、is_forced、rollback_version |

### 9.3 数据规则

- 所有数值必须有单位。
- 所有指标必须有适用年龄段。
- 按性别拆分的数据不能混合展示。
- 缺失值必须有说明字段。
- 来源不可用时，页面必须降级展示“来源待补充”，并允许反馈。
- 数据更新必须保留变更记录。

## 10. 医疗安全与文案规范

### 10.1 必须遵守

- 只能使用“参考范围”“建议关注”“如有疑问请咨询专业医生”等表达。
- 不使用“正常”“异常”“偏高”“偏低”“发育迟缓”等诊断式表达，除非来源明确且文案经过审核。
- 警示信号页面必须提示及时就医。
- 疫苗信息必须提示以当地接种单位和最新政策为准。
- 体检信息必须提示具体项目以当地儿保机构要求为准。

### 10.2 页面提示要求

| 页面 | 必须提示 |
| --- | --- |
| 首页 | 本工具仅供参考，不能替代专业医疗建议 |
| 生长指标详情 | 个体差异较大，如担心请咨询医生 |
| 疫苗详情 | 接种安排以当地接种单位为准 |
| 体检详情 | 体检项目以当地儿保机构要求为准 |
| 警示信号 | 出现明显异常或持续担忧时及时就医 |

### 11.3 离线与缓存

- MVP 可内置静态数据包，减少服务端依赖。
- 数据更新上线后，需要支持版本号对比。
- 更新失败时继续使用旧数据。
- 强制更新只用于严重数据错误或结构不兼容。

## 12. 状态与异常

| 状态 | 处理要求 |
| --- | --- |
| 无宝宝信息 | 允许手动选阶段，并提示可填写出生日期 |
| 出生日期超范围 | 提示产品适用于 0-3 岁参考 |
| 指标缺失 | 展示缺失说明和来源反馈入口 |
| 加载中 | 首页核心内容使用骨架屏 |
| 加载失败 | 提供重试和返回首页 |
| 搜索无结果 | 推荐热门词、分类入口或全部阶段 |
| 反馈提交失败 | 保留输入内容，允许重试 |
| 数据更新失败 | 保留旧版本，提示稍后重试 |

## 14. 验收清单

### 14.1 v1.0 MVP 必须通过

- 输入出生日期后能正确计算月龄和阶段。
- 不输入出生日期时也能手动选择阶段。
- 首页能展示当前阶段、体重、身高、头围和重点提醒。
- 多维对比页能按体重、身高、头围对比不同阶段数据。
- 多维对比页能按睡眠、喂养、发育对比不同阶段建议。
- 当前宝宝所在阶段在对比页中有明确高亮。
- 阶段详情能展示该阶段完整内容。
- 维度详情能按阶段展示内容。
- 数据说明页包含来源、版本、更新时间、适用范围和免责声明。
- 所有医疗相关页面都有参考提示。
- 缺失数据和仅有增长速度的数据不会被误展示为具体范围。

### 14.2 v1.1 增强版必须通过

- 搜索结果可跳转到具体内容。
- 无搜索结果时有替代入口。
- 来源依据能从关键字段进入。
- 反馈提交能带入上下文并返回编号。
- 数据版本更新失败时不会影响旧数据使用。

### 14.3 v1.2 传播版必须通过

- 成长卡内容准确反映当前阶段。
- 分享海报不包含误导性医疗判断。
- 分享链接或图片生成失败时有明确提示。

## 15. 待确认问题

- MVP 是否必须包含搜索，还是先用分类和阶段入口替代。
- 数据包是前端静态内置，还是服务端接口下发。
- 反馈是否已有后台处理机制。
- 原文摘录是否涉及版权或授权限制。
- 疫苗和体检内容是否需要按地区差异展示。
- 是否需要内容审核流程确认医疗安全文案。
